# PRD.md

## 1. 产品定位
Manus-lite 是一个单用户、本地优先的 AI Chat + Agent 客户端。

在传统聊天能力基础上，系统支持 Agent Runtime，允许模型在对话过程中自动判断是否调用工具、继续推理或输出最终结果。Agent 能够在多轮推理循环中完成复杂任务。

核心关键词：
- Chat
- Agent
- Tools
- Reasoning Loop

系统形态：
- 后端：Python + FastAPI，负责会话、消息、Provider 调度、记忆摘要、SSE
- 前端：Vue3 + Vite，负责会话 UI、流式展示、Think 展示、本地持久化

目标是以最小复杂度实现接近主流 AI 聊天产品的核心体验：**会话隔离 + 流式可见 + 记忆连续**。

## 2. 目标用户
- 个人开发者
- AI 应用学习者
- AI Agent 应用开发者
- 需要测试模型工具调用能力的用户
- 需要 GLM 云模型优先、并可扩展本地 Ollama/其他模型的用户

## 3. In Scope（当前范围）
1. 左侧会话列表与历史查看（支持删除）
2. 草稿态（刷新默认空白、`+ New` 进入草稿）
3. 首发消息后创建 pending 会话，首条流内容到达后再展示历史项
4. 标题规则：首问前 20 字（标准化后截断）
5. SSE 流式输出（start/thinking/delta/end/error）
6. 多会话并发隔离与会话级 loading
7. Think 缩略模块展示（400x400 逻辑视图缩小为 70x70）
8. Markdown 渲染（正文 + think）
9. IndexedDB 持久化（会话/消息/think）
10. 后端记忆机制（window + summary snapshot）
11. Provider 抽象：Ollama / GLM / Codex（默认 GLM `glm-4.7`）
12. 删除确认弹窗（Element Plus，暗色风格统一）
13. 会话项删除按钮在 `conversation-item` hover/focus-within 时显示，固定在标题行最右侧
14. 用户消息发送后需立即可见（乐观渲染），不得被异步历史加载覆盖
15. Think 仅保留“放大居中面板 / 回缩略”交互（去掉展开/收起）
16. 桌面缩略 Think 绝对定位在输入区：`top:-50px; left:30px`
17. `message-scroll` 增加基础底部留白 `80px`，避免缩略模块遮挡
18. 全局业务滚动区滚动条统一为高对比蓝色细条，悬停加亮
19. 输入框发送按钮左侧提供“深度思考”开关（控制 `enable_thinking`）
20. `message-scroll` 支持“自动滚底 / 手动暂停 / 一键恢复并滚底”
21. Agent Runtime：
    - 系统支持 Agent 推理循环
    - 模型在每轮输出中可返回 `thinking` / `tool_call` / `final_answer`
    - 系统根据模型输出自动执行工具，并将工具结果加入上下文后继续推理
    - 推理持续到模型输出最终结果或达到最大步数

## 4. Out of Scope（明确不做）
- 多用户登录与权限系统
- 文件上传/多模态
- 计费与配额系统
- 企业级审计合规模块
- 复杂 APM/Tracing 平台

## 5. 用户故事与验收标准

### Story A：我希望从空白草稿快速开始新对话
- Given 我刚刷新页面或点击 `+ New`
- When 我尚未发送第一条消息
- Then 右侧保持空白草稿态，输入框居中，左侧不新增历史项

验收：
- 默认不自动打开历史会话
- 草稿态输入框位置正确

### Story B：我希望首发后再展示历史会话，避免空会话污染
- Given 我在草稿态发送首问
- When 后端创建会话且流式开始
- Then 仅在收到第一条流内容后，左侧才显示该会话

验收：
- 无首条流内容时不显示该会话
- 会话标题为首问前 20 字

### Story C：我希望多会话并发不互相影响
- Given 会话 A 正在流式
- When 我切到 B 并继续发送
- Then A/B 各自流式与 loading 独立

验收：
- 不串流
- 回切 A 可继续看到增量

### Story D：我希望看到并管理模型 thinking 内容
- Given 模型返回 thinking 内容
- When 我查看当前会话
- Then think 在输入区上方以 70x70 缩略展示，点击后以 1:1 尺寸在 chat-panel 最高层级上下左右居中查看（非遮罩弹窗）

验收：
- think 与正文分离
- 两者都可 Markdown 渲染
- 缩略模块逻辑内容尺寸为 `400x400`，缩放后视觉尺寸 `70x70`
- 桌面端缩略定位：`top:-50px; left:30px`
- 点击缩略后，Think 同一内容视图在 chat-panel 最高层级上下左右居中展示（无遮罩层）
- 展开/收缩动画参考 macOS Genie Effect（神奇灯）风格

### Story E：我希望安全删除历史会话
- Given 我悬停某个会话项（conversation-item）
- When 我点击删除并确认
- Then 会话从列表与本地持久化移除，后端标记归档

验收：
- 删除按钮在 `conversation-item` hover/focus-within 或按钮 focus 时可见
- 弹窗风格与暗色主题一致
- 删除当前会话后回草稿态

### Story F：我希望发送后立即看到自己的问题
- Given 我在会话里输入并发送消息
- When 请求发出且流式响应尚未结束
- Then 用户消息应立即显示在消息区顶部顺序中，不等待 SSE 回包

验收：
- 首发消息在 200ms 内进入消息列表（不依赖后端回放）
- 并发流式或切换会话时，已发送用户消息不丢失不闪退
- 异步历史拉取不得覆盖本地未完成流式回合

### Story G：我希望 Think 区域仅保留放大和缩小，且不挡输入
- Given 模型正在输出 thinking 内容
- When thinking 文本持续增量到达
- Then Think 缩略模块保持可点击放大，并且不遮挡输入区可用性

验收：
- 去掉展开/收起逻辑，仅保留“放大居中面板 / 回缩略”
- 桌面端缩略显示不影响输入框点击、输入、选中与发送按钮点击
- 移动端（`<=980px`）隐藏缩略模块，输入框左侧显示放大入口（有 think 内容时显示）
- 放大面板首次打开必须自动滚动到底部
- thinking 未结束时，放大面板默认持续自动滚底；用户手动滚动后暂停自动滚底
- 放大面板底部提供“回到底部并恢复自动滚动”悬浮按钮

### Story H：我希望滚动条在全局业务区域统一且可见
- Given 我在会话列表、消息区、Think 区或放大区滚动
- When 鼠标进入滚动区域
- Then 我看到统一蓝色高对比细滚动条，悬停时加亮

验收：
- 覆盖所有业务滚动容器（会话列表、消息区、think 缩略内容、think 放大区内容等）
- 默认细条，hover/focus 明显加亮，active 再增强
- 非业务第三方弹层滚动条不被无差别重写

### Story I：我希望消息区流式输出时自动滚底且可手动接管
- Given assistant 正在持续流式输出
- When 我没有手动上滚消息区
- Then `message-scroll` 自动保持在底部，能实时看到最新输出

验收：
- 流式增量期间默认自动滚底
- 用户手动上滚后立即暂停自动滚底
- 暂停后在 `message-scroll` 右下角显示圆形悬浮按钮
- 点击按钮后恢复自动滚底并立即滚动到底部

### Story J：我希望按需开启/关闭深度思考模式
- Given 我准备发送问题
- When 我点击发送按钮左侧的“深度思考”开关
- Then 本次后续请求携带对应 `enable_thinking` 状态（开启/关闭）

验收：
- 开关位于输入框发送按钮左侧，视觉状态明确
- 开关状态变化后，后续流式请求 `enable_thinking` 与开关一致
- GLM 与 Ollama 都支持该开关输入；不支持时后端安全降级

## 6.1 稳健验收（必须通过）
1. 缩略模块不挡输入：
   - 缩略显示后，输入框任意位置可点击、可输入、可选中
   - 缩略模块外透明区域不得拦截 pointer 事件
2. 80px 留白可用性：
   - `message-scroll` 基础 `padding-bottom` 固定为 `80px`
   - 在 100% / 125% / 150% 缩放下，最后一条消息不被缩略模块遮挡
   - 用户上滚后，消息区“恢复自动滚底”按钮可见且点击后生效
3. 滚动条不污染：
   - 所有业务滚动区样式统一为蓝色高对比细条
   - Element Plus 等第三方非业务滚动区不被意外重写
4. 流式性能稳定：
   - 60s 连续 thinking 流式时输入打字无明显卡顿
   - 长 think（>=10k 字）可正常放大查看，滚动顺畅
## 7. 成功标准

### 功能指标
- 可稳定完成会话创建、聊天、流式、历史查看、删除
- 多会话并发下无串流/错写
- loading 状态与真实流状态一致

### 体验指标
- 首 token 可见时间可接受（本地模型条件）
- 滚动行为稳定（仅左栏与消息区滚动）
- Think 展示清晰且不干扰正文阅读
- 发送回显稳定：用户提问即时出现，无“只见 thinking 不见问题”的感知缺陷
- 消息跟随稳定：流式输出默认跟底，用户可随时手动接管并一键恢复

## 8. 失败与边界场景
1. Provider 不可用：`/health` 显示 false，聊天接口返回标准错误
2. 网络/CORS 错误：前端请求失败但界面不崩溃
3. 会话不存在：返回 404
4. SSE 中断：会话进入 error/done，不自动续传
5. 流式首次失败：pending 会话不进入历史列表
